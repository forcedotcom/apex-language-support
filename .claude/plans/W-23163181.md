### Phase 3 — ApexClientMiddleware type + default logging middleware

- `src/ApexClientMiddleware.ts` — `next`-based chain interface per spec (sendRequest/sendNotification/onRequest/onNotification, all optional, `next` continuation). JSON-RPC layer — NOT vscode-languageclient Middleware.
  - Each interceptor is modeled as an `Effect.fn`-wrapped continuation (named span, e.g. `"ApexClientMiddleware.sendRequest"`), so composition uses Effect combinators (the continuation is `next: () => Effect<...>`). Do NOT define interceptors as plain imperative callbacks that the core threads by hand.
- `src/middleware/loggingMiddleware.ts` — default observability middleware:
  - Logs method + direction via **`Effect.log`** (structured fields: `{ method, direction }`), then yields `next`. It does NOT accept a `Logger` parameter and never imports `vscode-jsonrpc` `Logger`.
  - Logger output is bound at the boundary by providing a Layer. The middleware itself stays logger-agnostic (SKILL.md:516-535). Two implementation options for the bridge — pick during build, document the choice:
    1. Reuse `EffectLspLoggerLive`/`EffectTestLoggerLive` re-exported from `@salesforce/apex-lsp-parser-ast`. Simplest, but adds a hard dep on the heavy parser-ast package (antlr4/apex-parser/protobuf) for a tiny layer.
    2. **Preferred:** add a small local `src/logging/EffectLspLoggerLayer.ts` (BSD header) modeled on `packages/apex-parser-ast/src/utils/EffectLspLoggerLayer.ts`, depending only on `effect` + shared `getLogger`/`getLogLevel`. Keeps the SDK lean (no parser-ast dep) and the bridge fully vscode-free. If chosen, drop `@salesforce/apex-lsp-parser-ast` from Phase 1 deps/refs.
  - If 4.1 needs to route SDK logs to a wire `Logger` the consumer supplies, that is done by composing a bridge Layer (modeled on `EffectLspLoggerLayer.ts`), NOT by passing a `Logger` into the middleware.
- **Composition is layer/Effect-based, not a mutable list.** The middleware *type* + default impl land here. Wiring/order is expressed as Effect composition (fold interceptors via Effect combinators), to be finalized in Concern-4. No `middlewares: ApexClientMiddleware[]` array with hand-rolled onion compose anywhere.
- Export both.

### Phase 4 — ApexClientCore skeleton + Concern 1 lifecycle

- `src/ApexClientCore.ts` — thin imperative shell over an Effect service body. **No mutable class fields.** All lifecycle state (`initialized`, `disposed`, registered middleware, cleanup) lives in `Ref`/`SubscriptionRef` + `Scope` inside the Effect, created once when the core is constructed and captured by the closures the public methods delegate to (SKILL.md:42).
  - **Construction model.** The core is built via an Effect (e.g. a factory `make(connection, opts)` returning `Effect<ApexClientCore, never, Scope>`), so cleanup is scoped. The public class wraps it: each method runs its inner Effect at the boundary (`Effect.runPromise(inner.pipe(Effect.provide(loggerLayer)))`) and exports `Promise`/plain types only — no Effect in exported signatures (SKILL.md:10).
  - **ctor / factory registers default handlers BEFORE messages flow.** Registers: default logging middleware (Phase 3) + default `findMissingArtifact` responder via `connection.onRequest('apex/findMissingArtifact', …)` returning `{ notFound: true }`.
    - **Precondition (doc-comment, load-bearing):** the passed `connection` MUST NOT be started/listening yet. Handlers are registered during core construction so they are in place before `initialize()` runs and before the server can send `apex/findMissingArtifact`. Adapter responsibility (2.3/4.1): create connection → build/construct Core (handlers register) → THEN start the connection. Mirrors `language-server.ts` which registers handlers before `client.start()`. If an adapter passes a live connection, registration races the initialize handshake — call this out so adapters order it correctly.
    - Literal `'apex/findMissingArtifact'` + inline `{ notFound: true }` is a deliberate temporary; replace w/ shared registry constant + `FindMissingArtifactResult` type in 3.1. Mark w/ TODO: replace in 1.1/3.1.
  - `use(mw: ApexClientMiddleware): Disposable` — append the middleware into a `Ref<ReadonlyArray<...>>` (not a plain mutable array field); the returned `Disposable` removes it from the Ref. Composition/order is via Effect combinators (Phase 3), not hand-rolled onion. Each registration also tracks cleanup so dispose tears it down (see below).
  - **Concern 1 lifecycle:**
    - `initialize(settings: ApexLanguageServerSettings, params?: Partial<InitializeParams>): Promise<InitializeResult>` — build `InitializeParams` w/ `initializationOptions = settings` (default `DEFAULT_APEX_SETTINGS`); then **strict sequential awaits per the LSP spec:** `const result = yield* Effect.promise(() => connection.sendRequest('initialize', params))` to obtain `InitializeResult`, **THEN** (only after the response resolves) `yield* Effect.promise(() => connection.sendNotification('initialized'))`. These are sequential — the `initialized` notification is sent strictly after the `initialize` response resolves, never fire-and-forget/parallel (LSP protocol requirement).
      - **Idempotency via `Effect.cached`**, not a boolean flag: wrap the initialize Effect so repeated calls return the same memoized `InitializeResult` and do NOT re-send `initialize`/`initialized` (the cached compute-once primitive is created at construction and stored in the captured state) (SKILL.md:42; Effect docs `Effect.cached`).
    - `shutdown(): Promise<void>` — send `shutdown` (request) then `exit` (notification), sequentially; idempotent (Ref/`Effect.cached`-guarded compute-once, not a flag).
    - `dispose(): void | Promise<void>` — driven by the construction `Scope`. **Ordering is explicit and load-bearing:** finalizers run LIFO, so register them such that middleware/handler cleanup (final log flush, unsubscribe, `onRequest` Disposables) runs FIRST and `connection.dispose()` (transport teardown) runs LAST. Use `Effect.addFinalizer` / scope finalization, NOT a hand-rolled disposable array (SKILL.md:420). Closing the scope = dispose; if the transport were torn down first, middleware cleanup that emits a final message would hit a closed transport — the LIFO/finalizer ordering prevents that. Mark dispose idempotent.
    - `isDisposed()` — reads the disposed `Ref`, not a class field.
  - Effect internals allowed; exported signatures = Promises/plain types only (no Effect in signatures, no Ref/Scope leaked).
- Export `ApexClientCore` (+ options type) from `index.ts`.
- Adapter delegation + the connection-not-started precondition + dispose-ordering assumption noted in doc-comments; wiring in 4.1.
