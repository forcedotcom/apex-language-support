# W-23163183 — 2.2 standard LSP pass-throughs + next-based middleware chain

## Context

- Builds on W-23163181 (foundation): `ApexClientCore`, `RpcConnection` port, `ApexClientMiddleware` type, `use()` registration, logging middleware all exist.
- Current gap: `use()` registers middlewares in a `Ref` but **nothing invokes the chain**. `connection.sendRequest` is called directly for lifecycle. No typed pass-through methods exist.
- This WI delivers:
  - **Concern 2**: typed wrappers for `textDocument/hover`, `textDocument/completion`, `textDocument/definition`, `textDocument/documentSymbol` + generic `request<P,R>(method, params)` escape hatch.
  - **Concern 4**: wire the `next`-based middleware chain so all outgoing requests/notifications flow through registered middlewares in registration order (onion). Both directions (outgoing sends + incoming handler dispatch).
- Effect boundary preserved: new public methods return `Promise`/plain types only.
- Dep addition: `vscode-languageserver-protocol` (types-only, already in monorepo via shared's `vscode-languageserver` dep) for `HoverParams`, `Hover`, `CompletionParams`, `CompletionList`, `CompletionItem`, `DefinitionParams`, `Definition`, `Location`, `LocationLink`, `DocumentSymbolParams`, `DocumentSymbol`, `SymbolInformation`.
- Lifecycle methods (`initialize`, `shutdown`) intentionally bypass the middleware chain — they are idempotent/memoized and run before/after the chain is meaningful.

## Design decisions

### D1 — Incoming handler registration model (addresses finding adversary:high re connection.onRequest)

**Problem**: `findMissingArtifact` registered at construction (130-143) via `connection.onRequest` with direct handler. Compose-incoming-chain cannot retroactively intercept already-registered handler. Plan cannot assume "chain intercepts handlers registered before it exists."

**Decision**: option (b) — intercept at the `connection.onRequest` call-site, not at dispatch time. Instead of wrapping individual handler forks after-the-fact, the core creates a **wrapping registration helper** (`registerIncomingRequest`) that:

1. Accepts a method + raw handler.
2. Registers a *composed* handler on the connection: `connection.onRequest(method, (params) => composeIncomingRequestChain(middlewareRef.current, rawHandler, method, params))`.
3. The composed handler reads `middlewareRef` at invocation time (not registration time), so middlewares registered after construction are still observed.

All incoming handler registrations in `makeCore` (including `findMissingArtifact`) go through this helper. The existing `findMissingArtifact` registration is rewritten to use it. No "handlers-before-traffic" precondition is violated — the handler is still registered at construction before traffic flows; the only change is that the handler function itself folds through the middleware chain at call time (lazily reading the current middleware set).

This preserves:
- The documented precondition (W-23163181:78) that handlers are registered before traffic.
- Late-bound middleware observability — a middleware added via `use()` after construction still intercepts `findMissingArtifact`.

The separate `forkLog` in the current `findMissingArtifact` handler is removed — logging middleware now observes the request naturally via the chain.

### D2 — Notification chain execution semantics (addresses finding adversary:high re void return)

**Problem**: `sendNotification`/`onNotification` return void, must call `next` synchronously. Async work before `next` = unobserved transformation (race).

**Decision**: notification chains strictly synchronous, sequential, no-await. Contract:

1. A notification middleware MUST call `next(params)` synchronously (before returning). If it performs async side-effects (telemetry upload, metric flush), those are fire-and-forget AFTER calling `next` — they do not gate the chain.
2. A middleware MAY transform `params` synchronously before passing to `next`. This is the only supported transformation path for notifications.
3. A middleware MAY short-circuit by NOT calling `next` (suppress the notification).
4. The compose function enforces this by design: it builds a synchronous call chain (no Promises in the fold). Each middleware's `next` is the composed tail of remaining middlewares + the innermost `connection.sendNotification`. The entire chain executes in a single synchronous tick.

Restriction documented in `ApexClientMiddleware` JSDoc: "Notification middlewares execute synchronously. Async side-effects (logging, telemetry) must be fire-and-forget after calling `next`; they MUST NOT gate param transformation or `next` invocation." This is already the de-facto pattern in `loggingMiddleware.sendNotification` (which calls `forkLog` fire-and-forget then `next(params)` synchronously).

This eliminates race conditions: synchronous execution guarantees that a params-transforming middleware's output is observed by the next middleware before `connection.sendNotification` fires.

## Phases (each = 1 commit)

### Phase 1 — middleware composition engine

- New `src/middleware/composeMiddleware.ts`:
  - `composeRequestChain<P, R>(middlewares, sendFn, method, params): Promise<R>` — folds middlewares into onion; innermost `next` calls `sendFn` (`connection.sendRequest` for outgoing / raw handler for incoming). Registration order = execution order (first = outermost).
  - `composeNotificationChain<P>(middlewares, sendFn, method, params): void` — synchronous fold for notifications (outgoing or incoming). Innermost `next` calls the provided `sendFn`. Entire chain executes in one synchronous tick. No Promises in the fold — enforcing D2.
  - Both functions accept a `direction: 'outgoing' | 'incoming'` parameter that selects which middleware hook to invoke (`sendRequest`/`sendNotification` vs `onRequest`/`onNotification`).
  - The fold reads from a snapshot `ReadonlyArray<ApexClientMiddleware>` (caller reads `middlewareRef` at invocation time per D1).
- Internal (not exported from package index) — consumed only by the core.

commit: `feat(apex-lsp-client): middleware composition engine (next-based onion) - W-23163183`

### Phase 2 — wire chain into ApexClientCore

- Modify `apexClientCore.ts`:
  - Add private `sendRequest<R>(method, params)` helper: reads `middlewareRef` snapshot, calls `composeRequestChain` with direction `'outgoing'` and `connection.sendRequest` as the terminal `sendFn`.
  - Add private `sendNotification(method, params)` helper: reads `middlewareRef` snapshot, calls `composeNotificationChain` with direction `'outgoing'` and `connection.sendNotification` as the terminal `sendFn`. Returns `void` (synchronous per D2).
  - Add private `registerIncomingRequest(method, rawHandler)` helper (D1): registers on the connection a composed handler that reads `middlewareRef` at call time (late-bound), folds through `composeRequestChain` with direction `'incoming'` and `rawHandler` as the terminal `sendFn`. Returns the `Disposable` from `connection.onRequest`.
  - Add private `registerIncomingNotification(method, rawHandler)` helper (D1): same pattern for notifications, using `composeNotificationChain` with direction `'incoming'`.
  - **Rewrite `findMissingArtifact` registration** to use `registerIncomingRequest`. The raw handler is `(_params) => ({ notFound: true })`. The separate `Runtime.runFork(runtime)(logMiddlewareEvent(...))` call is removed — logging middleware now observes the request via the chain naturally (D1).
  - The connection precondition (handlers registered before traffic) remains satisfied: `registerIncomingRequest` is called during `makeCore`, before the connection is started.
- Expose on `ApexClientCore` class:
  - `request<R>(method: string, params?: unknown): Promise<R>` — generic escape hatch.
  - `notify(method: string, params?: unknown): void` — generic notification pass-through. Returns `void` (synchronous chain per D2).
- `CoreHandle` interface updated with new methods.

commit: `feat(apex-lsp-client): wire middleware chain into core sends/receives - W-23163183`

### Phase 3 — typed standard LSP pass-throughs

- New `src/lspPassThroughs.ts` — LSP param/result types re-exported from `vscode-languageserver-protocol` for SDK consumers; keeps import paths clean.
- Add to `ApexClientCore`:
  - `hover(params: HoverParams): Promise<Hover | null>`
  - `completion(params: CompletionParams): Promise<CompletionList | CompletionItem[] | null>`
  - `definition(params: DefinitionParams): Promise<Definition | LocationLink[] | null>`
  - `documentSymbol(params: DocumentSymbolParams): Promise<DocumentSymbol[] | SymbolInformation[] | null>`
- Each delegates to `this.request(METHOD, params)` with the correct method string (`textDocument/hover`, etc.).
- Add `vscode-languageserver-protocol` to `package.json` dependencies.
- Re-export relevant param/result types from `src/index.ts`.

commit: `feat(apex-lsp-client): typed hover/completion/definition/documentSymbol pass-throughs - W-23163183`
